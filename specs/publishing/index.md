# Result Publishing

Publishing (committing, pushing, and creating PRs) is handled by the agent as a pipeline step, not as a built-in system action. This makes publishing recoverable — if the agent forgets to commit, encounters a push rejection, or hits a GitHub API error, it can diagnose and fix the problem or ask the user for help.

## How it works

The final step in a pipeline is typically an agent call with a publishing prompt (e.g., `.feliz/prompts/publish.md`). The prompt instructs the agent to:

1. Verify all changes are committed (if not, stage and commit them)
2. Push the branch to the remote
3. Create a PR via the GitHub/GitLab API
4. Post the PR link and summary back to Linear

The agent has access to git CLI and GitHub API tools (via MCP or environment variables like `GITHUB_TOKEN`) to perform these actions.

## Publishing prompt

The default publishing prompt (generated by `feliz project add`) instructs the agent:

```markdown
# Publish

You are finalizing work on {{ issue.identifier }}: {{ issue.title }}.

## Steps

1. **Verify commits**: Check `git status`. If there are uncommitted changes, stage and commit them with a descriptive message referencing {{ issue.identifier }}.

2. **Push**: Push the branch `feliz/{{ issue.identifier }}` to origin. If the push is rejected (e.g., upstream changes), rebase on {{ project.base_branch }} and retry.

3. **Create PR**:
   - Title: `[{{ issue.identifier }}] {{ issue.title }}`
   - Body: Include a link to the Linear issue, a summary of changes, files changed, and test results.
   - Base branch: `{{ project.base_branch }}`

4. **Report**: Output the PR URL so Feliz can post it back to Linear.

If any step fails, describe what went wrong and attempt to fix it. If you cannot fix it, explain the problem clearly so the user can help.
```

## Linear Status Updates

After the publishing step completes, Feliz (the orchestrator) posts a comment on the Linear issue with:
- PR link (extracted from agent output)
- Summary of changes
- Run duration and token usage

Feliz updates the Linear issue state according to the configured mapping (default: -> "In Review").

If the publishing step fails, Feliz posts the failure details and the agent's diagnosis to Linear, allowing the user to intervene.

## Agent tools for publishing

The agent needs access to:
- **git CLI** — already available in the worktree
- **GitHub/GitLab API** — via `GITHUB_TOKEN` / `GITLAB_TOKEN` environment variable passed to the agent, or via MCP tools
- **Linear API** — optionally, for posting comments directly (or Feliz handles this post-step)

These are provided through the agent's environment (`AgentRunParams.env`).
